# 🎯 تدفق تسجيل التاجر الجديد - مرحلتين

## 📋 نظرة عامة
تم إعادة هيكلة عملية تسجيل التاجر لتكون على **مرحلتين منفصلتين** بدلاً من مرحلة واحدة طويلة:

### ✅ المزايا:
- **تبسيط التسجيل**: المستخدم لا يملأ بيانات كثيرة دفعة واحدة
- **حل مشكلة RLS**: بعد تأكيد البريد، المستخدم يكون مسجل دخول بشكل طبيعي
- **تجربة مستخدم أفضل**: خطوات واضحة ومنطقية
- **تقليل الأخطاء**: لو حصل خطأ في بيانات المتجر، الحساب موجود بالفعل

## 🔐 Row-Level Security (RLS)

### السياسات الحالية (لا تحتاج تعديل):
✅ **merchants_insert_policy**: `WITH CHECK (auth.uid() = id)`  
   - يسمح للمستخدم المسجل دخول بإنشاء سجله الخاص

✅ **stores_insert_policy**: `WITH CHECK (auth.uid() = merchant_id)`  
   - يسمح للتاجر بإنشاء متاجر مرتبطة بحسابه

**لماذا تعمل الآن؟**
- بعد تأكيد البريد، المستخدم **مسجل دخول** (`auth.uid()` موجود)
- لا نحتاج تسجيل دخول مؤقت أو حلول معقدة
- `auth.uid() = id` و `auth.uid() = merchant_id` يعملان بشكل طبيعي
